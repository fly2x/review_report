Integer overflow in BuildPolyInput leading to heap buffer overflow
crypto/modes/src/modes_gcm_siv.c:166-176
Reviewers: GEMINI, CLAUDE, CODEX 置信度: 可信 
static uint32_t Pad16Len(uint32_t len)
{
    return (len + GCM_BLOCKSIZE - 1u) & ~(GCM_BLOCKSIZE - 1u);
}

static int32_t BuildPolyInput(struct ModesGcmSivCtx *ctx, const uint8_t *plaintext, uint32_t ptLen,
    uint8_t **polyBuf, uint32_t *polyLen)
{
    uint32_t padAad = Pad16Len(ctx->aadLen);
    uint32_t padPt = Pad16Len(ptLen);
    uint32_t total = padAad + padPt + GCM_BLOCKSIZE;
    uint8_t *buf = BSL_SAL_Malloc(total);
Issue: Both Pad16Len() and BuildPolyInput() have unchecked arithmetic that can overflow for large inputs. Pad16Len() at line 168 can wrap when len is close to UINT32_MAX. BuildPolyInput() at line 176 calculates total = padAad + padPt + GCM_BLOCKSIZE without overflow checks. This causes malloc to allocate a smaller buffer than needed, and subsequent memcpy_s operations use wrapped destMax values, leading to heap buffer overflow.
Fix:
static int32_t Pad16Len(uint32_t len, uint32_t *padded)
{
    if (len > UINT32_MAX - GCM_BLOCKSIZE + 1) {
        BSL_ERR_PUSH_ERROR(CRYPT_INVALID_ARG);
        return CRYPT_INVALID_ARG;
    }
    *padded = (len + GCM_BLOCKSIZE - 1u) & ~(GCM_BLOCKSIZE - 1u);
    return CRYPT_SUCCESS;
}

    uint8_t **polyBuf, uint32_t *polyLen)
{
    uint32_t padAad;
    uint32_t padPt;

Issue: MODES_GCM_SIV_DupCtx uses BSL_SAL_Dump to copy the entire struct including the outPtr field. The outPtr points to caller-allocated buffer. If the duplicated context is used for another encryption/decryption operation, it will write ciphertext to the original caller's buffer when GET_TAG is called, causing heap corruption since the original buffer was sized for the original message only.
