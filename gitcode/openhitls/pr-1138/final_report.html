BN_MontExp no longer creates temporary optimizer when opt is NULL
crypto/bn/src/bn_mont.c:287-295
Reviewers: CODEX 置信度: 可信 
static int32_t MontExp(BN_BigNum *r, const BN_BigNum *a, const BN_BigNum *e, BN_Mont *mont,
    BN_Optimizer *opt, bool consttime)
{
    int32_t ret = MontParaCheck(r, a, e, mont);
    if (ret != CRYPT_SUCCESS) {
        return ret;
    }

    return MontExpCore(r, a, e, mont, opt, consttime);
}
Issue: The implementation that created a temporary optimizer when opt == NULL was removed. MontExp now forwards NULL directly to MontExpCore, which immediately uses the optimizer. The API signature did not change, so callers that previously relied on the automatic fallback will now crash with a null dereference.
Fix:
static int32_t MontExp(BN_BigNum *r, const BN_BigNum *a, const BN_BigNum *e, BN_Mont *mont,
    BN_Optimizer *opt, bool consttime)
{
    int32_t ret = MontParaCheck(r, a, e, mont);
    if (ret != CRYPT_SUCCESS) {
        return ret;
    }

    bool newOpt = (opt == NULL);
    if (newOpt) {
        opt = BN_OptimizerCreate();
        if (opt == NULL) {
            BSL_ERR_PUSH_ERROR(CRYPT_MEM_ALLOC_FAIL);
            return CRYPT_MEM_ALLOC_FAIL;
        }
    }

    ret = MontExpCore(r, a, e, mont, opt, consttime);
    if (newOpt) {
        BN_OptimizerDestroy(opt);
    }
    return ret;
}

Issue: r1 and r2 are allocated with bits size instead of bits>>1. Since r1 = p - 1 and r2 = q - 1 where p and q are roughly bits/2 sized primes, they fit in bits/2 + 1 bits. Using bits size is wasteful for a bin size optimization PR.
