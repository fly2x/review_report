        HIAE_Stream_Encrypt(tmpState, out, in, inLen);

TITLE: FlushPendingData always processes full 16-byte block
PROBLEM: The function passes sizeof(block) (16 bytes) to HIAE_Stream_ProcAD even when only partial data (ctx->dataBufLen bytes) is buffered. While the zero-padding is correct for the algorithm, this is inconsistent with how the cipher's CommitPendingMsg handles partial blocks and could be confusing.
CODE:
```c
static int32_t FlushPendingData(PQCP_HIAE_MacCtx *ctx)
{
    uint8_t block[HIAE_BLOCK_SIZE] = {0};
    if (ctx->dataBufLen == 0) {
        return PQCP_SUCCESS;
    }
    if (memcpy_s(block, sizeof(block), ctx->dataBuf, ctx->dataBufLen) != EOK) {
        return PQCP_INVALID_ARG;
    }
    HIAE_Stream_ProcAD(ctx->state, block, sizeof(block));
    ctx->dataBufLen = 0;
    return PQCP_SUCCESS;
}
```
FIX:
Add a comment explaining that zero-padding is intentional per the HiAE spec, or consider passing the actual length and letting HIAE_Stream_ProcAD handle padding (but verify the spec first).
